## Overview

Unit of Measure (UOM) conversions enable flexible inventory transactions. You can receive stock in one unit (e.g., boxes), store it in another (e.g., individual units), and sell it in yet another (e.g., grams).

<Info>
  **Example**: Receive salmon in 5kg boxes, store as kg, transfer in grams, and track costing in kg.
</Info>

## UOM Structure

### UnitOfMeasure (Base Units)

Base units are the foundation of the conversion system.

<ParamField body="code" type="string" required>
  Unique UOM code (max 20 chars, typically uppercase)
  
  **Examples**: `KG`, `G`, `L`, `ML`, `UNIT`, `BOX`, `PACK`
</ParamField>

<ParamField body="name" type="string" required>
  Full UOM name (max 100 chars)
  
  **Examples**: `Kilogram`, `Gram`, `Liter`, `Milliliter`, `Unit`, `Box`, `Pack`
</ParamField>

<ParamField body="symbol" type="string">
  Display symbol (max 10 chars)
  
  **Examples**: `kg`, `g`, `L`, `mL`, `u`, `bx`, `pk`
</ParamField>

<ParamField body="precision" type="integer" default="2">
  Decimal places for display (0-6)
  
  **Examples**: 0 for whole units, 2 for currency, 4 for scientific
</ParamField>

<ParamField body="is_decimal" type="boolean" default="true">
  Whether fractional quantities are allowed
  
  - `true`: Can use 1.5 kg, 0.25 L
  - `false`: Only whole numbers (e.g., 1 unit, 5 boxes)
</ParamField>

<ParamField body="is_active" type="boolean" default="true">
  Active status (inactive UOMs hidden from selection)
</ParamField>

### UomConversion (Conversion Rules)

Defines directed conversion factors between two UOMs.

<ParamField body="from_uom_id" type="integer" required>
  Source UOM ID
</ParamField>

<ParamField body="to_uom_id" type="integer" required>
  Target UOM ID
</ParamField>

<ParamField body="factor" type="decimal(6)" required>
  Multiplication factor: `target_qty = source_qty × factor`
  
  **Examples**: 
  - KG → G: factor = 1000 (1 kg = 1000 g)
  - BOX → UNIT: factor = 24 (1 box = 24 units)
  - G → KG: factor = 0.001 (1 g = 0.001 kg)
</ParamField>

<ParamField body="tolerance" type="decimal(4)" default="0.00">
  Acceptable variance percentage for physical counts
  
  **Example**: tolerance = 2.0 means ±2% variance is acceptable
</ParamField>

<ParamField body="is_active" type="boolean" default="true">
  Active status (inactive conversions not used)
</ParamField>

## Conversion Calculation

### Forward Conversion

Convert from source UOM to target UOM:

```
target_quantity = source_quantity × factor
```

**Example**: 5 kg → g
- factor = 1000
- result = 5 × 1000 = 5000 g

### Reverse Conversion

Convert back (inverse):

```
source_quantity = target_quantity ÷ factor
```

**Example**: 5000 g → kg
- factor = 1000 (kg → g)
- result = 5000 ÷ 1000 = 5 kg

<Info>
  Always create **bidirectional conversions** for seamless transactions in both directions.
</Info>

### Chained Conversions

For multi-step conversions (e.g., BOX → PACK → UNIT):

```
final_qty = original_qty × factor1 × factor2
```

**Example**: 2 boxes → units (BOX → PACK → UNIT)
- BOX → PACK: factor = 10 (1 box = 10 packs)
- PACK → UNIT: factor = 50 (1 pack = 50 units)
- Result: 2 × 10 × 50 = 1000 units

<Warning>
  Current system supports **single-step conversions only**. For chained conversions, create direct conversion rules or pre-calculate factors.
</Warning>

## Tolerance in Physical Counts

The `tolerance` field defines acceptable variance in physical counts:

```
variance_pct = abs((actual - expected) / expected) × 100
acceptable = (variance_pct <= tolerance)
```

**Example**: Tolerance = 2.0% (2.0)

| Expected | Actual | Variance | Acceptable? |
|----------|--------|----------|-------------|
| 100 kg | 102 kg | 2.0% | ✓ Yes |
| 100 kg | 103 kg | 3.0% | ✗ No |
| 50 L | 49.5 L | 1.0% | ✓ Yes |
| 50 L | 48 L | 4.0% | ✗ No |

<Info>
  Tolerance is used during physical inventory counts to flag significant discrepancies that require investigation.
</Info>

## Validation Rules

### UOM Creation

<Tabs>
  <Tab title="Required Fields">
    - `code` (unique, max 20, uppercase recommended)
    - `name` (max 100)
  </Tab>
  <Tab title="Optional Fields">
    - `symbol` (max 10, default: same as code)
    - `precision` (0-6, default: 2)
    - `is_decimal` (default: true)
    - `is_active` (default: true)
  </Tab>
  <Tab title="Constraints">
    - `code` must be unique
    - `precision` must be between 0 and 6
  </Tab>
</Tabs>

### Conversion Creation

<Tabs>
  <Tab title="Required Fields">
    - `from_uom_id` (must exist)
    - `to_uom_id` (must exist)
    - `factor` (> 0)
  </Tab>
  <Tab title="Optional Fields">
    - `tolerance` (≥ 0, default: 0)
    - `is_active` (default: true)
  </Tab>
  <Tab title="Constraints">
    - `from_uom_id` ≠ `to_uom_id`
    - `factor` must be positive
    - `tolerance` must be non-negative
    - Unique: (from_uom_id, to_uom_id)
  </Tab>
</Tabs>

### Item Type UOM Rules

| Item Type | UOM Conversions | Reasoning |
|-----------|----------------|-----------|
| INSUMO | Multiple allowed | Flexible receiving, storing, and consumption |
| PRODUCTO | Base only (discouraged) | Finished goods typically have fixed packaging |
| ACTIVO | Base only (1:1) | Assets counted as discrete units |

## Common Workflows

### Workflow 1: Set Up Weight System

<Steps>
  <Step title="Create Base UOMs">
    POST `/api/v1/units-of-measure` for KG, G, LB, OZ
  </Step>
  <Step title="Create Metric Conversions">
    KG↔G bidirectional conversions
  </Step>
  <Step title="Create Imperial Conversions">
    LB↔OZ bidirectional conversions
  </Step>
  <Step title="Create Cross-System Conversions">
    KG↔LB and G↔OZ for mixed-unit scenarios
  </Step>
  <Step title="Set Tolerances">
    Assign 1-2% tolerance for physical counts
  </Step>
</Steps>

### Workflow 2: Set Up Item-Specific Packaging

<Steps>
  <Step title="Create Generic Count UOMs">
    UNIT, PACK, BOX, CASE (if not exists)
  </Step>
  <Step title="Define Item Packaging">
    For "Soy Sauce": 1 CASE = 4 BOX, 1 BOX = 6 BTL
  </Step>
  <Step title="Create Item Conversions">
    POST conversions specific to this item's packaging
  </Step>
  <Step title="Test with Sample Transaction">
    Register opening balance in CASE, verify conversion to BTL
  </Step>
</Steps>

### Workflow 3: Handle Multi-UOM Receipt

<Steps>
  <Step title="Vendor Ships in Mixed UOMs">
    Receipt contains: 10 boxes + 50 individual units
  </Step>
  <Step title="Create Multi-Line Movement">
    Line 1: 10 BOX → converts to base  
    Line 2: 50 UNIT → base UOM
  </Step>
  <Step title="System Converts Each Line">
    Total stock in base UOM = (10 × box_factor) + 50
  </Step>
  <Step title="Stock Updated">
    Single stock record updated with total base quantity
  </Step>
</Steps>

## Database Schema

### units_of_measure Table

| Column | Type | Constraints |
|--------|------|-------------|
| id | bigint | PK, auto-increment |
| code | varchar(20) | UNIQUE, NOT NULL |
| name | varchar(100) | NOT NULL |
| symbol | varchar(10) | NULLABLE |
| precision | smallint | DEFAULT 2, CHECK (0-6) |
| is_decimal | boolean | DEFAULT true |
| is_active | boolean | DEFAULT true |
| meta | jsonb | DEFAULT '{}' |
| created_at | timestamp | |
| updated_at | timestamp | |

**Indexes**:
- `idx_uom_code` on (code) UNIQUE
- `idx_uom_active` on (is_active)

### uom_conversions Table

| Column | Type | Constraints |
|--------|------|-------------|
| id | bigint | PK, auto-increment |
| from_uom_id | bigint | FK → units_of_measure.id, NOT NULL |
| to_uom_id | bigint | FK → units_of_measure.id, NOT NULL |
| factor | decimal(12,6) | NOT NULL, CHECK (> 0) |
| tolerance | decimal(10,4) | DEFAULT 0, CHECK (>= 0) |
| is_active | boolean | DEFAULT true |
| meta | jsonb | DEFAULT '{}' |
| created_at | timestamp | |
| updated_at | timestamp | |

**Indexes**:
- `idx_uom_conversion_from` on (from_uom_id)
- `idx_uom_conversion_to` on (to_uom_id)
- `idx_uom_conversion_lookup` on (from_uom_id, to_uom_id, is_active)

**Constraints**:
- Unique: (from_uom_id, to_uom_id)
- Check: `from_uom_id != to_uom_id`
